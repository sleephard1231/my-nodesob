# Long-term Memory for AI Agents

> 来源：B站直播分享
> 时长：80分钟 | 语言：中文（PPT英文）

---

## 🧠 Agent Memory 是什么

演讲者提出的定义：

> **Agent Memory = 跨时间的意图和状态的管理系统**

Agent 有三个面向：
- 面向人类用户的部分
- 面向大语言模型的部分
- Agent 自身的 runtime（运行环境）

---

## ❗ 为什么 Memory 重要

### 没有 Memory 的后果

> 用户留存率 **45%** — 超过一半的用户在使用一次后就流失了。

不是因为功能不好，而是用户每次回来都要**重新建立上下文**。

### 用户的真实需求

用户最讨厌的是：
- 每次回来要重新搭建上下文
- 已经说过目标、做过选择、推进到一半 — 全丢了
- **"不要再让我重复 setup"** — 这是 Memory 的第一价值

---

## 🔍 三个核心问题

### 1. 用户意图天生不完整、不清晰

> "把这个给我整理一下" — 这种表达在大模型时代很常见，但其实是**非常模糊的指令**

用户不是不想说清楚，是**用户自己也很难表达清楚**。你不能要求用户每次都说精准的 prompt。

### 2. 状态通过交互动态涌现

用户和 Agent 交互的过程中，不断涌现新的：
- 创意
- 需求
- 决策
- 需要调用的工具

这些**不是预设的**，而是在对话中"涌现"出来的。

### 3. 没有 Memory 就无法长期运行

经过多轮对话或行动后，Agent 必须记住：
- 之前做了什么
- 现在进行到什么阶段
- 下一步要做什么

否则无法达到长期运行的状态。

---

## 🏗️ 两种理解视角

### 视角一：ChatGPT（产品端）

- ChatGPT 早期最大的问题：**没有跨 session 的记忆**
- 用户在 2025 年秋冬季开始高频使用后，发现这个痛点非常明显
- ChatGPT 后来做了改进：分支对话、记忆功能
- ChatGPT 有两种 Memory：
  - **Safe Memory**（长期有效的事实和偏好）
  - 上下文窗口记忆

### 视角二：Cursor / Claude Code（工程端）

- 命令行工具端有 **resume** 功能 — 可以把整个上下文 resume 回来
- 偏向 chat-based 的产品，**连续性**是个人用户非常需要的

## 📊 关键数据

| 指标 | 数据 |
|------|------|
| 无 Memory 的用户留存 | 45% |
| 用户流失率 | ~50%（一半用户使用完就流失） |

---

## 💡 核心观点

1. **Memory 不是「记住我是谁」**，而是 **"不要再让我重复 setup"**
2. 用户意图是不完整的，Agent 需要在不完整的输入下工作
3. Agent 的状态是在交互中动态涌现的，不是预先定义的
4. 没有 Memory，Agent 只能做一次性的任务，无法长期运行
5. 无论是 AI 硬件产品、软件产品还是 AI Agent 开发者，都要解决 Memory 问题
