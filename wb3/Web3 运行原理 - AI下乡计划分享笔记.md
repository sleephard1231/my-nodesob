## 一、钱包与个人主权

### 1.1 核心概念

| 概念 | 类比 | 说明 |
|------|------|------|
| **私钥（Private Key）** | 银行账号密码 | 一串极长字符串，能授权各种链上操作 |
| **助记词（Seed Phrase）** | 私钥的备份 | 12/24 个单词，通过派生路径可生成无数私钥（一对多关系） |
| **公钥 / 地址** | 银行账号 | 由私钥计算得出，取后 20 字节（40 字符）作为链上地址 |

### 1.2 钱包的本质

- **钱包不存资产**，资产的状态在链上。
- 钱包只是帮你保存私钥、发起签名、调用链上数据。
- **不同钱包可以导入同一个助记词**（比如 Rabby → MetaMask），不像银行 A 的账号不能在银行 B 用。

### 1.3 个人主权的起点

- 点一下就能生成钱包，**不需要任何人批准，也不需要身份认证**。
- 生成后即可收币、管理资金。
- 这对亚非拉等金融体系不完善的地区意义重大——他们没有银行，但可以有钱包。

### 1.4 安全性要点

- 这不是普通密码——**私钥/助记词不能改**。区块链不认识你，只认签名。
- 一旦泄露（钓鱼、中木马、截图发网上），钱包就不安全了，**必须立刻转移资产**。
- 丢失助记词 = 丢失所有派生地址的控制权。

## 二、交易与签名

### 2.1 交易的本质

一笔交易不是只有转账，它是**一段授权网络执行某件事的数据**。

从第一性原理拆解交易的四要素：

1. **你要做什么** — 转账 / 合约调用 / 投票 / mint NFT
2. **手续费（Gas Fee）** — 为存储和计算付费
3. **Nonce（序号）** — 防止交易重放攻击（每次执行后 +1）
4. **签名** — 验证这笔交易确实是你发出的

### 2.2 数字签名流程

```
发送方（你的钱包）
  │  私钥 + 交易信息 → Hash → 生成签名
  ▼
广播到区块链网络（通过 RPC）
  │  发出三个信息：原始交易 + 发送地址 + 签名
  ▼
接收方节点
  │  用 EC Recover 算法：签名 + 原始交易 → 反推出签名者地址
  │  对比发送地址，一致则验证通过
  ▼
确认是你的意图
```

- 签名**不一定非要上链**，可以用来做身份验证。
- 量子计算如果攻破底层哈希算法，能碰撞出相同的签名——整个区块链就垮了。这是 Web3 未来的挑战。

### 2.3 Gas Fee（汽油费）

- **作用 1：过滤垃圾交易** — 发一次请求要付钱，就不会随便发。21 年 NFT 最火的时候，一次 mint 要 150 刀 Gas，根本没人发垃圾交易。
- **作用 2：激励出块者** — 分两部分：
  - **Base Fee** → 直接销毁，实现通缩
  - **Priority Fee（小费）** → 给 Proposer/Validator，让交易优先打包
- **本质：谁占用资源谁付费**，驱动整个链运转。

所有交易数据都可以在 Etherscan 看到，包括 From/To、金额、手续费、Gas Price 等。

## 三、去中心化网络的运行

### 3.1 从交易到出块的完整链路

```
钱包签名 → RPC 广播 → Mempool（内存池排队）
  → Builder 排序打包 → Validator 验证 → 出块上链
```

### 3.2 各环节说明

| 组件 | 角色 | 说明 |
|------|------|------|
| **RPC** | 桥梁 | 传统互联网 ↔ 去中心化网络的桥，提供 HTTP/WebSocket 服务 |
| **Mempool** | 内存池 | 交易排队等待打包的地方，在节点的内存里 |
| **Builder** | 打包者 | 从 Mempool 里挑 Gas 给得多的交易优先打包 |
| **Validator** | 验证者 | 3 万多个验证者一起见证，确认块没问题后才上链 |

### 3.3 区块确认时间

以太坊（PoS）：
- 12 秒一个 slot（区块提议）
- **≈12 分钟达到安全确认**（Finality）
- 12 分钟内的块状态是 `Safe`，超过才是 `Finalized`
- 每个块引用上一个块的 Hash，链越长，篡改成本越高

### 3.4 共识机制对比

| | PoW（比特币） | PoS（以太坊） |
|--|--|--|
| 方式 | 算力竞赛，解哈希难题 | 质押 32 ETH 成为验证者 |
| 特点 | 耗电，随机性 | 节能，随机选 Proposer |
| 风险 | 算力集中化 | 质押集中化 |
| 验证 | 任何人都可验证 | 3 万+ 验证者共同见证 |

PoW：做数学题，谁先解出来谁记账 → 获得出块奖励
PoS：质押 → 被选为 Proposer → 出块 → 其他验证者见证 → 如果作恶则被罚没（Slashing）

### 3.5 RPC 的中心化问题

- RPC 节点本质上是一些公司跑的服务（Infura、Alchemy 等）
- 用户实际能连的 RPC 也就二三十个
- 这些公司如果挂了 / 拒绝服务，你就用不了链
- 解决方案：自建节点（但成本高，同步数据量大）

## 四、智能合约

### 4.1 本质

- **一段写在链上、可被交易触发的代码**
- 以 Bytecode 形式存储，在 EVM（以太坊虚拟机）中运行
- 不可篡改（部署后代码不能变）
- 执行历史全部写入区块，**公开透明可查**

### 4.2 信任的转移

传统世界：信任律师、政府、中间人 → 签合同
Web3：**信任代码、信任共识**

> Code is Law

一旦合约部署 + 资产转入，规则一定会被执行。这消除了信任成本——买房找律师的时间资金成本大家都懂。

### 4.3 可升级合约（代理模式）

虽然合约不可篡改，但可以用**代理模式**实现升级：

```
用户 → 调用 Proxy 合约（地址不变）
         ↓
Proxy 代理转发到 Implementation 合约（实际逻辑）
         ↓
升级时：部署新 Implementation 合约，在 Proxy 里更新地址
         ↓
用户无感知，还是调用同一个 Proxy 地址
```

- 相当于一个**指针**，指向不同的实现合约
- 一些 DeFi 协议还会加**时间锁（Timelock）**——升级后 7 天才生效
  - 这样用户能看到逻辑改了，有反应时间（撤资等）
  - 历史上黑客在假期攻击，因为有时间锁保护才被及时发现拦截

## 五、协议升级（以太坊硬分叉）

### 5.1 升级流程

```
论坛讨论（Ethereum Magicians）
  → AllCoreDevs 会议讨论
  → 形成 EIP 文档（Eth Improvement Proposal）
  → 各个客户端（Geth、Nethermind 等）实现代码
  → 开启 Feature Flag
  → 主网升级
```

### 5.2 硬分叉

- 每一次以太坊主网升级都是硬分叉——创建一条新链
- 新链改了共识规则（比如增加新字段）
- 旧链理论上可以继续运行，但**没人维护了就没人用了**
- 除非共识有重大分歧（如 ETH → PoW 分叉出 ETC），否则基本就是全链平滑迁移
- 用户无感知，升级期间可能会有短暂不稳定，但链会自己修复

### 5.3 客户端多样性

- 执行层客户端：Geth（Go）、Nethermind（C#）、Reth（Rust）等
- 共识层客户端：Prysm、Lighthouse 等
- 好处：一个客户端出 Bug 不会拖垮全网
- 历史上确实有客户端代码写错导致硬分叉的情况，但因为其他客户端正常，网络没受影响

## 六、Web3 的关键特性

| 特性 | 说明 | 局限性 |
|------|------|--------|
| **去中心化** | 任何人可创建钱包，节点全球分布 | RPC 层面有中心化问题 |
| **无需许可** | 任何人都可使用、读写 | 但需付 Gas 防垃圾 |
| **抗审查** | 节点全球分布，软件多样 | 国家级别封禁仍有影响 |
| **开源开放** | 所有客户端开源，交易透明可验证 | — |
| **隐私性** | 能看到地址的余额，但不知道是谁 | — |
| **可组合性** | 合约之间可相互调用、组合 | — |

## 七、Web3 的跨学科本质

Brose 老师的总结：Web3 不是单纯的技术，也不是投机炒作，而是 **三门学科的交汇**：

```
密码学（Cryptography）
  ├── 签名算法、哈希、ECDSA
  └── 保护所有权，可验证
经济学（Economics）
  ├── PoS、质押、罚没、Gas 机制
  └── 激励协调参与者
社会学（Sociology）
  ├── 去中心化治理、共识、自我主权
  └── 数字世界控制权的重新分配
```

> 密码学保护所有权，经济激励协调参与者，社会共识决定规则——**一环不行，整个网络就玩不转。**

## 八、Q&A 要点补充

### 8.1 Validator 的经济逻辑
- 全网有上百万个 Validator，每个质押 32 ETH
- 12 秒一个 Slot，随机选一个 Proposer 出块
- 出块奖励来自 Gas Fee（Base Fee 销毁，Priority Fee 给 Proposer）
- Validator 作恶会被 Slashing 惩罚

### 8.2 节点之间如何发现
- 依赖 **Boot Node**（硬编码在客户端源码里的初始节点）
- 新节点启动 → 连 Boot Node → 获取附近节点列表 → 相互广播传播
- 形成一个去中心化的 P2P 网络

### 8.3 RPC 怎么盈利
- 读是免费的（因为成本低，服务器闲时资源就够了）
- 写操作成本高（链上存储和执行消耗 Gas），需要付费
- 高频/机构用户：自建节点或买付费 RPC 服务（更稳定、更快）
- 做链上数据标签/分析的团队：自建节点全量同步，存到中心化数据库再提供服务
